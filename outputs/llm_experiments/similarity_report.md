## 💡 API 환경에서 100% 결정론성(재현성) 확보가 어려운 이유

### 1. ❓ 지금의 API 방식(OpenRouter 등)에서는 왜 100% 결정론성이 불가능한가요?
사용자께서 seed=42, temperature=0으로 동일하게 설정하고 동일한 질문을 던져도 매번 출력이 바뀌는 이유는 "물리 하드웨어의 동적 분산 처리 환경" 때문입니다.

#### ① GPU 병렬 연산의 극미세 실수(Float) 오차
- LLM은 수천억 개의 실수를 GPU 행렬 곱셈 연산으로 처리합니다. 컴퓨터의 실수 연산은 계산하는 순서에 따라 결과가 미세하게 다를 수 있습니다. (예: (A + B) + C vs A + (B + C)의 끝자리 오차 발생)
- 수많은 사용자의 요청을 초당 수천 개씩 동시에 빠르게 처리해야 하므로, API 서버의 GPU들은 성능 극대화를 위해 매 순간 연산 처리 순서를 바꿉니다. 이 과정에서 발생하는 아주 미세한 연산 편차가 눈덩이처럼 굴러가 최종 결과물(단어 선택이나 괄호 표기 등)을 바꾸어 놓습니다.

#### ② 클라우드 분산 서빙의 불안정성
- OpenRouter 같은 API 게이트웨이는 전 세계에 분산된 수많은 서버 인프라로 요청을 전달합니다.
- 1초 전에 보낸 요청은 미국 서버의 A100 GPU가 처리하고, 2초 후에 보낸 요청은 아시아 서버의 H100 GPU가 처리할 수 있습니다.
- 하드웨어 칩셋의 종류, 동시 접속자 수(배치 크기), 심지어 API 업체의 시스템 메모리 상태에 따라 매번 물리 실행 환경이 달라지기 때문에 소프트웨어적으로 시드를 고정해도 무용지물이 됩니다.

### 2. 🤖 OpenAI나 Claude 같은 대형 독점 모델로 바꾸면 해결될까요?
"어느 정도 완화는 되지만, 근본적인 100% 결정론성 확보는 여전히 불가능합니다."

- **OpenAI (GPT-4o 등)**나 Anthropic (Claude 3.5 Sonnet) 같은 독점 대형 API는 자신들의 단일 인프라를 타이트하게 관리하므로, OpenRouter 등의 다중 분산 환경보다 일치율이 훨씬 높습니다 (재현성 95~98% 수준).
- 실제로 OpenAI는 하드웨어 연산 매칭 정보를 식별해 주는 system_fingerprint를 API 응답에 실어 결정론성을 보장하려 노력합니다.
- 하지만 그들 역시 트래픽이 몰리는 피크 시간대나 백엔드 인프라 업데이트 시에는 **출력이 가끔 튀는 현상(Non-determinism)**이 지속 보고되고 있으며, 이는 전 세계 LLM 엔지니어들의 공통된 고민거리입니다.

### 3. 🖥️ "온프레미스 고정 인프라"는 왜 완벽한 해결책이 되나요?
- 정의: 외부 클라우드 API를 쓰지 않고, **우리 회사 소유의 단일 GPU 서버에 직접 LLM(예: DeepSeek 오픈소스 버전)을 얹어서 전용으로 서빙(호스팅)**하는 방식입니다.
- 해결 원리:
   1. 동적 배치 제거: 우리 시스템만 단독으로 호출하므로 동시 접속자에 의한 연산 순서 꼬임이 없습니다.
   2. 하드웨어 단일화: 연산을 처리하는 물리 GPU 장치가 매번 100% 동일합니다.
   3. 엔진 강제 고정: vLLM 등의 서빙 엔진 설정 시, 속도를 소폭 희생하는 대신 병렬 행렬 연산의 처리 순서 자체를 강제로 일치시키는 물리 옵션(deterministic=True)을 적용할 수 있습니다. 이 경우 바이트 레벨까지 100% 일치하는 결과를 출력하게 됩니다.

### 4. 실무 PoC 차원의 현실적인 제안
따라서 현실적인 비즈니스 배포 관점에서는 다음 중 하나의 정책적 의사결정이 필요합니다.

1. 인프라 관점 (비용 투자):
   - 보안 및 재현성이 100% 보장되어야 하는 프로덕션 단계에서는 자체 GPU 인프라(온프레미스/전용 클라우드 인스턴스)에 모델을 직접 탑재해 통제합니다.
2. 소프트웨어 관점 (비즈니스 룰 타협):
   - "글자 하나 안 틀리고 매번 똑같아야 한다"는 엄격한 결정론성 제약을 다소 유연하게 풀고, **"의미론적 유사도 검사(대안 B)를 통해 95% 이상 일치하면 합격으로 치고, 이탈한 극소수 케이스는 우리의 override 시스템 사후 보정 로직으로 감싼다"**는 구조를 기본으로 정립하고 활용하는 것입니다.

오케스트레이터 및 팀원분들과 머지 가이드라인을 논의할 때, 이러한 API의 기술적 배경지식을 바탕으로 결정론 검증의 수용 한계치(Fuzzy 기준)를 합의하시는 것이 PoC를 성공적으로 머징하는 지름길입니다.
