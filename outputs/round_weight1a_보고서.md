## 1. dedup_routing.yaml 스키마 동작

사람이 작성한 도면4 섹션을 코드는 읽기만 했다:

```yaml
도면4:
  기둥:
    SC1: {count_from: "1층 구조평면도", spec_from: "1층 구조평면도"}
    SC2: {count_from: "1층 구조평면도", spec_from: "1층 구조평면도"}
```

- `dedup_loader` 가 평면 `DedupRoute` 2행으로 로드 (메모·주석 블록은 무시).
- `count_from`/`spec_from` 빈값·None → ValueError, 같은 (도면,부호) 중복 → ValueError 로 사람 실수를 잡는다.
- 길이는 dedup yaml 이 아니라 `length_routing.yaml`(별도 관리)에서 가져왔다.

**스키마 검토 결과 — 도면4 에서 부족했던 필드: 없음.** 자명한 케이스라 count_from =
spec_from = "1층 구조평면도" 한 시트로 충분했다. (1b 라운드 확장 후보는 §5 참조.)

## 3. 단위중량 계산식 vs KS 표 차이

baseline-1 `unit_weight_calc` 는 H형강 4세그먼트 단면적
`A = B·tf·2 + tw·(H − tf·2)` 에 밀도 7,850 kg/m³ 를 곱한다. 모서리 곡률(필렛)을
무시하므로 KS D 3502 표값 대비 **약 -2~3% 작게** 나온다. 도면4 합계 -2.9% 도 이
범위 안이다. **이 차이는 알려진 한계이며 이번 라운드 fix 범위가 아니다** — 단면적
식 정확화(필렛 반영 또는 KS 표 룩업)는 별도 라운드.

## 5. 알려진 한계 / 다음 라운드(1b)

### 이번 라운드에서 다루지 않은 것
- **도면1·2·3·5** — dedup_routing.yaml 확장 + 동일 흐름 적용 (1b)
- **보 부재(SG·SB)** — 1b 이후 별도 라운드
- **단면적 식 정확화** — KS 표 대비 -2~3%, 별도 라운드
- **도면2 SC1·SC2 카운트** — 익명 블록 내 분리 TEXT 데이터 한계(known-fail 2종)

### 1b 라운드 입력 — 스키마 확장 후보
도면4 는 자명해 필드 부족이 없었으나, 다른 도면 확장 시 필요할 항목:
- **`by_section` (동·구역 분리)** — 도면1 주심도 vs 부호도, 1동·2동 중복. dedup 결정이 동별로 갈림.
- **`count_override`** — 도면2 처럼 측정 데이터 한계로 정답지 값을 강제해야 하는 경우 격리용 필드.
- **`length_from` 명시 여부** — 현재 길이는 length_routing.yaml 이 전담. dedup yaml 에 길이 시트까지 명시할지 1b 에서 판단.
- **보 섹션(`보:`)** — 한 시트=보카운트+기둥길이 양립 한계와 맞물림(기존 누적 한계 3종 중 1종).
