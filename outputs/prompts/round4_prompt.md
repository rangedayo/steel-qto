# 라운드 4 — 도면2 height 필터 적용 + SC 누락 정량화

## 현재 상황 (라운드 3 결과)

도면1: 22/22 정확 일치, 총합 331=331, 평균오차 0% ✅

도면2 진단으로 두 개의 다른 문제가 드러남:
- **문제 ① SB/SG 과다 (+4)**: height 301 작은 글자 오탐 → height 필터로 해결 가능
- **문제 ② SC1·SC2 대량 누락 (−12)**: 정답 SC1=10·SC2=4인데 시스템이 각 1개만 검출. 인계 메모의 발견("도면2 SC는 익명 블록 + 분리 TEXT, 즉 'SC'와 '1'이 따로 저장")이 실측으로 확증됨

도면2 총합: 현재 36 vs 44 (−8)
- height 필터만 적용 시: SB/SG 정확 + SC 더 악화 → **약 −16 예상**

- 분리 TEXT 결합 룰을 counter.py에 추가하지 말 것 (인계 메모의 "룰 무한 패치" 함정 회피)

## 작업 1 — 도면2 height 필터 적용

`config/symbol_rules.yaml`의 `text_height_filter`에서 도면2 항목 수정:

```yaml
text_height_filter:
  도면1:
    min_height: 177       # 라운드 3 적용 완료
  도면2:
    min_height: 302       # height 301(작은글자)은 제외, 400(큰글자)만 카운트
  도면4:
    min_height: null      # 라운드 5 이상에서
```

임계값 결정 근거: 도면2 height는 400(큰)·301(작은) 두 값으로 100% 갈림 → 302로 잡으면 301은 잘리고 400은 통과.

counter.py, baseline.py, test_regression.py는 라운드 3에서 이미 설정 파일을 읽도록 수정됨. **추가 코드 변경 없이 yaml 한 줄만** 변경.

## 작업 4 — 도면2 SC 누락 원인 직접 조사 (룰 추가는 X, 조사만)

새 스크립트 `tests/investigate_도면2_sc.py` 작성. 목적: 도면2의 SC 부재가 DXF의 어디에 어떻게 저장돼 있는지 *진단*. 룰 추가나 counter.py 변경은 절대 금지.

조사할 것:

### 4-A. SC 부재의 위치 단서 찾기
- 도면2를 ezdxf로 열어서 modelspace의 모든 INSERT 엔티티 순회
- 각 INSERT의 블록 정의로 들어가서, 그 안에 포함된 TEXT/MTEXT/ATTRIB/ATTDEF의 내용을 모두 수집
- 'SC'로 시작하거나 'C1', 'C2' 같은 짧은 텍스트와 인접한 'S' 텍스트가 있는 블록을 찾아 출력

### 4-B. 분리 TEXT 패턴 정량화
한 블록 안에 다음 조합이 있는 블록 개수 카운트:
- `'S'` + `'C'` + 숫자 → SC 부재 후보
- `'S'` + `'C1'` 또는 `'SC' + '1'` 같은 다른 분할
- 위 분할이 같은 INSERT 블록 내부에 있는지, 별개 엔티티인지

각 분할 패턴별로:
- 블록 개수
- 블록명 (익명 블록인지 명명 블록인지)
- INSERT 위치 좌표 (대략적 분포)

### 4-C. 다른 부호의 누락 가능성 점검
SC만 이런 분리 인코딩을 쓰는지, 아니면 다른 부호(MC, SG 등)도 일부 그런지 확인:
- 정답지 도면2의 각 부호별로, 검출 누락이 큰 부호 추가 점검
- 단순히 검출이 안 된 부호 = 누락 후보 ≠ 모두 분리 TEXT 문제 (다른 원인일 수도)

### 4-D. 결과 출력
표 형태로:
```
부호  정답  검출(필터적용)  누락추정  원인후보
SC1   10    0              10       분리TEXT(블록 N개)
SC2   4     0              4        분리TEXT(블록 M개)
...
```

**이 조사는 진단 데이터만 만든다. counter.py에 절대 어떤 룰도 추가하지 말 것.**

## 작업 6 — 멘토 보고용 리포트 작성

`round4_도면2_누락_정량화.md` 파일 작성. 다음 구조로:

### A. 라운드 1~4 진척 요약 (한 페이지)
- 도면1: 27% → 32% → **100%** (라운드 1 → 2 → 3·4)
- 도면2: 라운드 3 진단으로 두 문제 발견, 라운드 4 height 필터 적용 결과
- 도면4: 미착수 (사전 점검만)

### B. 도면2의 핵심 발견
- height 필터로 SB/SG는 해결됨 (구체 수치)
- SC 누락은 height와 무관한 별개 문제: 분리 TEXT 인코딩
- 작업 4 조사 결과 표 포함 (SC 부재가 어떤 블록 구조에 있는지)

### C. 의사결정 갈래 (멘토에게 묻고 싶은 것)

| 방향 | 작업량 | 위험 | 도면2 5% 가능성 |
|---|---|---|---|
| A. 룰 기반 분리 TEXT 결합 | 중 | 도면1 회귀 위험, 도면3·5에서 또 다른 분리 패턴 가능 | 높음 |
| B. 비전 보조 (인계 메모의 nanobanana 등) | 대 | 외부 의존, 1단계 본질에서 멀어짐 | 중 |
| C. 도면2는 미해결로 두고 2단계(규격 매칭) 진입 | 소 | 도면2 평가 못 받음, 도면1만으로 1단계 완료 처리 | 0 (도면2 포기) |

각 갈래의 작업 분량, 위험, 회사 평가 기준(5% 오차) 충족 가능성을 사실관계로 정리. 의견은 멘토 판단에 맡기되 사용자의 권고는 명시할 수 있음.

### D. 인계 메모와의 충돌 명시
인계 메모 원문 인용:
> "방식 3 (익명 블록 + 분리 TEXT), 방식 6 (LINE으로 그린 글자)은 룰 기반 패치 무한 추가 위험으로 방치 결정. 향후 VLM 보조 후보."

라운드 3 실측이 이 발견을 확증했음을 보고. 갈래 A로 가려면 메모의 방치 결정을 뒤집는 것이고, 그건 멘토 결정 사항임을 명시.

### E. 도면4 사전 점검 결과 (작업 5)
도면4도 같은 문제 패턴인지 미리 짚음. 누락 패턴 부호와 과다 패턴 부호 분류 표.
