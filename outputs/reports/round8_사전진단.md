**핵심 발견 — 도면1·2 와 height 구조 정반대:**
- 도면1·2: 본체가 큰 글자, 작은 글자는 주석/오탐 → `min_height` 위쪽 컷이 정답
- 도면3: 본체(221.8)가 가장 작은 글자, 일람표(275.6)가 더 큰 글자, 보 부호(337.5)가 더 큼 → "큰 갭 위쪽" 알고리즘이 추천한 `min_height=338` 은 본체를 다 죽임

**라운드 6 D항(신호 1 자동화 구조적 불가) 와 같은 함정.** `text_height_filter` 추천 알고리즘이 도면3 에서 작동 안 함. 도면3 에는 `min_height: null` 이 맞음.

## 작업 2 진행을 위한 옵션 (멘토 결정 필요)

### 옵션 1 — 신호 3 임계값 낮추기
`auto_policy_params.spec_pattern_threshold`: 0.3 → 0.2 (또는 더 낮게)
- 도면3 기둥: 부호 4개 × 0.2 = 0.8 → 규격 1개 ≥ 0.8 → 신호 3 ON → C1 PASS
- **부작용 위험**: 도면1·2·4 자동 정책 판정이 흔들릴 수 있음. 라운드 6 베이스라인 깨질 가능성.
- 검증 필요: 도면1·2·4 에서 임계값 낮춰도 자동 판정이 같은 결과인지.

### 옵션 2 — `policy_override` 로 도면3 신호 3 강제 ON (권장)
`config/symbol_rules.yaml` 의 `policy_override.도면3: { exclude_table_regions: true, exclude_with_spec: true }`
- 도면3 만 강제, 다른 도면 영향 0
- 라운드 6 도입한 비상 수단의 정당한 활용 (자동 판단이 못 잡는 케이스)
- 가장 안전한 길

### 옵션 3 — height 필터로 규격 텍스트 분리
도면3 만 `min_height: 222` 정도 (본체 221.8 살리고 규격 225.2 컷)
- 본체만 살리고 규격·일람표·큰글자 다 컷
- **부작용**: 일람표(275.6) 텍스트가 height 필터로 사라져 신호 2 자동 검출이 OFF 로 바뀜 → 일람표 4종 카운트 안 빠짐 → C1·C2·C3·C4 각 +1 오차 (모두 FAIL 가능성)
- 비추천

### 옵션 4 — 작업 2 보류, 도면3 제외
