#### 사람2 — 스키마 설계

회원가입 페이지에 가면 "이메일은 필수, @가 들어가야 함", "비밀번호는 8자 이상", "전화번호는 숫자만" 같은 규칙이 자동으로 검사된다. 잘못 입력하면 빨간 글씨로 "이메일 형식이 아닙니다" 뜸.

**그 규칙을 코드로 박아놓은 게 `routing.schema.json`이고, 그 규칙대로 자동 검사하는 도구가 `검증 코드`**

좀 더 구체적으로 설명하자면,

지금 yaml은 이렇게 생겼음 ↓↓

```yaml
도면4:
  기둥:
    SC1:
      count_from: "1층 구조평면도"
      spec_from: "1층 구조평면도"
```

LLM이 만들면 가끔 이런 오류를 낼 수 있다. ↓↓

```yaml
도면4:
  기둥:
    SC1:
      count_form: "1층 구조평면도"     # 오타! form → from
      spec_from: ""                    # 빈 문자열!
```

이걸 사람이 눈으로 잡으려면 5장 × 여러 부호 다 봐야 해서 힘들다. **스키마가 있으면 명령어 한 번에 자동으로 잡아준다.**

```markdown
$ python validate.py config/dedup_routing.yaml
❌ 도면4 > 기둥 > SC1: "count_form" 필드는 허용되지 않음 (혹시 "count_from" 오타?)
❌ 도면4 > 기둥 > SC1 > spec_from: 빈 문자열은 허용되지 않음
```

**즉, "yaml이 잘 만들어졌는지 자동 채점하는 도구"를 만드는 게 사람2의 일이다.**

#### **사람4 — 승인 코드로 짜기**

**"검토 게이트가 들어간 메인 스크립트를 코드로 짠다"**

가이드의 사람4 산출물 4단계를 풀어보면:

1. 사람3의 LLM이 yaml 초안을 뱉어냄
2. 사람2의 스키마로 그 yaml 형식이 맞는지 자동 검증
3. **검토자(사람)에게 yaml을 보여주고 "OK?" 묻는 단계를 코드에 넣음** — 이게 "검토 게이트"
4. OK 받으면 weight_pipeline 돌려서 CSV 출력

사람4는 "LLM이 실수했을 수도 있으니 사람이 한 번 보고 통과시켜야 다음으로 진행되는 흐름"을 파이썬 코드로 구현하는 사람이다. 이게 메인 스크립트 안에 박혀서 매번 자동 실행됨.
