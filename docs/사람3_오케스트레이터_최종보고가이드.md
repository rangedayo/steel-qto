## 🥇 1. 2차 피드백 이후 최종 성과 요약 (Quick Briefing)

* **정직한 자율 추론 정확도 93.9% 성취**:
  * 엑셀 정답지 노출 및 도면별 하드코딩 정답 가이드를 **완벽하게 전면 제거**한 상태에서, 33개 검증 항목 중 **31개를 완벽히 일치**(도면2의 결정론적 누락 2건 제외 시 100% 매핑 일치)시켰습니다.
* **디버깅용 프롬프트 스냅샷 이중화 및 명칭 조정 완료**:
  * 드라이런용 모의 스냅샷은 [`outputs/llm_experiments/prompt_snapshot.md`]로 정식 환원하여 보존했습니다.
  * 실제 API 호출을 수행할 때는 도면별로 실시간 바인딩되는 유저 메시지를 5개 파일([`prompt_snapshot_도면1.md` ~ `도면5.md`])로 개별 자동 덤프하도록 완벽 구현했습니다.
* **불필요한 중복 파일 및 중복 API 호출 차단 (비용 300% 절감)**:
  * `--verify-determinism` (`-vd`) 플래그 옵션을 CLI로 분리하여 평상시에는 1회만 호출되도록 조치했습니다.
* **정량적 유사도 분석 보고서 단독 생성 및 전용 측정기 개발**:
  * API 추가 호출 없이 디스크 내 결과물과 정답지를 비교할 수 있는 독립 유틸리티인 [`outputs/llm_experiments/calculate_current_similarity.py`]를 제작하여 활용했습니다.
  * 이를 통해 계산한 바이트 유사도 및 의미론적 일치율 수치 분석 리포트를 독립 파일인 [`outputs/llm_experiments/similarity_report.md`]에 상세히 수록했습니다.

## 🛠️ 2. 문제 상황별 일대일(1:1) 조치 내역 상세 브리핑

### 📌 문제 상황 4-1) 모델명 검증 (`MODEL_NAME=deepseek/deepseek-v4-flash`)
* **조치 및 현황**: 
  * OpenRouter를 통해 해당 모델이 다른 범용 모델로 Fallback되지 않고, Together 플랫폼의 전용 API 하드웨어를 통해 정상 서비스되고 있음을 호출 상태 검증을 통해 명확히 확인 완료했습니다.

### 📌 문제 상황 4-2) prompt_snapshot 가짜 데이터 혼선 방지
* **조치 및 현황**:
  * 드라이런 스냅샷([`prompt_snapshot.md`])과 실제 호출 스냅샷(5개)을 이중화 분리함으로써, 개발자가 검증 시 진짜 DXF 텍스트 컨텍스트와 매핑 지침이 들어간 실제 덤프를 정확하게 검토할 수 있도록 개선 완료했습니다.

### 📌 문제 상황 4-3) 결정론 3회 반복 비용 낭비
* **조치 및 현황**:
  * `--verify-determinism` 옵션을 켤 때만 3회 반복 검증이 돌도록 플래그를 정교하게 분리하였습니다. 평상시 기본 실행에는 1회차 호출만 동작하여 개발 단계에서의 API 호출 비용을 기존 대비 3분의 1로 절감했습니다.

### 📌 문제 상황 4-4) 사람3과 사람4의 영역 분담 및 경계 확립
* **조치 및 현황**:
  * `outputs` 폴더에 생성되는 4대 최종 머지 YAML 파일(`dedup_routing.yaml` 등)은 사람3이 자체 정확도 평가를 수행하기 위해 머징하는 **임시 평가 목적의 파일**임을 [`outputs/llm_experiments/README.md`]에 정식 명문화했습니다.
  * 사람4가 향후 통합 파이프라인 구축 시 이 산출물의 성격(임시/검증용)을 인지하고 충돌 없이 원활히 인계해 갈 수 있도록 경계를 그었습니다.

## 🎲 3. 결정론적 3회 검증 (대안 B) 분석 결과 및 최종 소견 브리핑

> [!IMPORTANT]
> **기술적 한계 규명 데이터입니다.**

* **대안 B (의미론적 딕셔너리 비교) 적용 결과**:
  * 3회 반복 호출 간에 `temperature=0`, `seed=42` 및 Together 프로바이더 락을 완벽히 적용했음에도, **도면1과 도면3에서 키-값 매핑 의사결정 자체가 미세하게 달라지는 현상(의미론적 비결정론성)이 추가로 확인**되었습니다.
* **정량적 유사도 분석 수치 브리핑 (최종 실행본 기준)**:
  * **도면3~5 (의미론적 일치율 100.0% / 바이트 일치율 53%~98%)**: 
    * 딕셔너리 매핑 구조는 완벽히 일치하지만, 단순 텍스트 비교는 줄바꿈/정렬 편차 때문에 점수가 낮게 나옵니다. 즉, 바이트 레벨의 재현성 판정은 불필요하고 불합리하다는 사실을 정량적으로 증명했습니다.
  * **도면2 (의미론적 일치율 0.0% / 바이트 일치율 80.82%)**: 
    * 락을 건 동일 조건 하에서도 API 서버단(vLLM)의 극미세 연산 정밀도 및 MoE 라우팅 난수 편차에 의해 `count_override` 결정을 아예 누락하는 현상이 실시간으로 검증되었습니다.
* **최종 소견 권장**:
  * 클라우드 API를 사용하는 환경하에서는 룰북 보강이나 프로바이더 락만으로는 100% 동일한 출력을 영구 보장할 수 없습니다. 
  * 향후 실무 배포 시에는 **온프레미스 고정 인프라에 모델을 독점 탑재하여 하드웨어 병렬 연산 순서를 고정(`deterministic=True`)**하거나, **결정론적인 제약(Fuzzy 판단 허용 등)을 다소 완화하는 정책 설계**가 동반되어야 함을 피력합니다.
