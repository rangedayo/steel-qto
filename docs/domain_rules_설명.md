# domain_rules.md / example.yaml 사용방법

작성일: 2026-06-02

---

## 1. 목적

이 문서는 사람1 산출물인 `domain_rules.md` 와 `example.yaml` 을 팀이 어떻게 사용해야 하는지 설명한다.

사람1 산출물의 역할은 다음과 같다.

- `domain_rules.md`: LLM이 도면을 읽을 때 따라야 할 도메인 규칙서
- `example.yaml`: LLM이 따라 쓸 수 있는 YAML 예시

즉 `domain_rules.md` 는 판단 기준이고, `example.yaml` 은 출력 형식의 본보기다.

---

## 2. 파일 위치

- 규칙서: [domain_rules.md](./domain_rules.md)
- 예시 YAML: [example.yaml](./example.yaml)

---

## 3. 누가 어떻게 쓰는가

### 3.1 사람3의 사용 방법

사람3은 LLM 프롬프트를 만들 때 이 두 파일을 같이 사용한다.

- `domain_rules.md` 로 LLM에게 도면 해석 규칙을 알려 준다
- `example.yaml` 로 LLM에게 출력 형식 예시를 보여 준다

즉 사람3은 아래 흐름으로 사용하면 된다.

1. `domain_rules.md` 를 읽고 프롬프트의 규칙 설명 부분을 만든다
2. `example.yaml` 을 few-shot 예시 또는 출력 예시로 넣는다
3. 새 도면에 대해 같은 방식의 YAML을 생성하게 한다

### 3.2 사람4의 사용 방법

사람4는 이 문서를 검증 기준과 검토 기준으로 사용한다.

- `domain_rules.md` 로 YAML 내용이 도메인 규칙에 맞는지 본다
- `example.yaml` 로 생성된 YAML의 구조와 표현 방식이 자연스러운지 비교한다

즉 사람4는 아래 흐름으로 사용하면 된다.

1. LLM이 만든 YAML을 받는다
2. `routing.schema.json` 으로 형식 검증을 한다
3. `domain_rules.md` 로 내용상 판단이 맞는지 검토한다
4. `example.yaml` 과 비교해 구조가 크게 벗어나지 않는지 본다
5. 통과하면 적산 파이프라인으로 넘긴다

### 3.3 사람1 본인의 사용 방법

사람1은 규칙서와 예시 YAML을 함께 유지보수해야 한다.

- 규칙이 바뀌면 `domain_rules.md` 를 먼저 수정한다
- 출력 예시가 바뀌면 `example.yaml` 도 함께 맞춘다

즉 규칙서와 예시 YAML은 따로 놀면 안 되고 항상 같은 기준을 반영해야 한다.

## 4. domain_rules.md는 어떻게 읽어야 하는가

`domain_rules.md` 는 코드 문서가 아니라 도메인 규칙서다.
따라서 아래 순서로 읽는 것이 좋다.

1. 용어 정의
2. 어떤 표기를 기둥 부호로 인정하는지
3. 본체와 일람표를 어떻게 구분하는지
4. 기둥 적산을 어떤 순서로 하는지
5. 예외 처리 원칙은 무엇인지

중요한 점은 이 문서가 `정답 값`을 주는 문서가 아니라 `어떻게 판단해야 하는지`를 주는 문서라는 것이다.

예를 들어 이 문서에서는 다음을 알려 준다.

- 기둥 개수는 평면 시트에서 부호별로 산정한다
- 기둥 형상만 있고 부호가 없으면 개수 산정 대상이 아니다
- 길이는 단면도 계열 시트에서 부호별로 산정한다
- 최종 적산은 부호별 총길이를 구한 뒤 같은 규격끼리 합산한다

즉 `domain_rules.md` 는 LLM이 도면을 읽을 때의 판단 순서를 고정하는 문서다.

---

## 5. example.yaml은 어떻게 써야 하는가

`example.yaml` 은 정답 데이터 전체가 아니라 `YAML 구조의 모범 예시`다.

예를 들어 아래와 같은 점을 보여 주기 위해 사용한다.

- `by_section` 을 어떻게 쓰는가
- `skip` 과 `skip_reason` 을 어떻게 쓰는가
- 각 부호 아래에 `count_from`, `spec_from` 을 어떻게 쓰는가

즉 `example.yaml` 의 역할은 다음과 같다.

- LLM이 출력 형식을 따라 쓰게 하는 예시
- 사람이 생성 결과를 비교할 수 있는 기준 예시
- 스키마와 문서가 실제로 어떤 모양을 기대하는지 보여 주는 예시

주의할 점은 `example.yaml` 이 모든 경우를 다 보여 주는 것은 아니라는 점이다.
예를 들어 `count_override` 같은 예외는 별도 사례에서 추가 예시가 필요할 수 있다.

## 6. 실제 사용 흐름

실제 팀 작업에서는 보통 아래 순서로 사용한다.

1. 사람1이 `domain_rules.md` 와 `example.yaml` 을 제공한다
2. 사람2가 이를 참고해 `routing.schema.json` 을 설계한다
3. 사람3이 규칙서와 예시 YAML을 프롬프트에 넣어 LLM 출력을 만든다
4. 사람4가 스키마 검증과 사람 검토를 거쳐 적산 파이프라인에 연결한다

즉 사람1 산출물은 사람2, 사람3, 사람4가 모두 참고하는 공통 기준 문서다.

---

## 7. 사람3 프롬프트에 넣을 때의 사용 예

사람3은 보통 아래처럼 사용할 수 있다.

- 규칙 설명 자료로 `domain_rules.md` 첨부
- 출력 예시로 `example.yaml` 첨부
- 요청 문장에는 `새 도면을 읽고 example.yaml 형식으로 YAML을 작성하라`는 식으로 지시

즉 `domain_rules.md` 는 `무엇을 어떻게 판단할지`를 알려 주고,
`example.yaml` 은 `어떤 모양으로 써야 하는지`를 알려 준다.

---

## 8. 사람4 검토 시 확인할 점

사람4는 아래를 같이 봐야 한다.

- YAML이 스키마 형식에 맞는가
- `count_from` 이 실제로 개수 산정 기준 시트인가
- `spec_from` 이 실제로 규격 판독 기준 시트인가
- `skip` 과 `count_override` 사용 이유가 규칙서와 맞는가
- 부호만 있고 규격, 기준 시트, 섹션 정보가 빠지지 않았는가

즉 스키마 검증은 형식 검사이고, `domain_rules.md` 검토는 내용 검사다.

---

## 9. 주의사항

- `domain_rules.md` 는 구현 코드 설명서가 아니다
- `example.yaml` 은 전체 프로젝트 모든 예외를 다 담은 완전한 데이터셋이 아니다
- 규칙서와 예시 YAML은 항상 같은 기준을 반영해야 한다
- 문서와 예시가 서로 어긋나면 사람3, 사람4가 혼란을 겪는다

## 10. 한 줄 요약
