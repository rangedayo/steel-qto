# 사람3 산출물 검토

`run_experiment.py` 자동화 흐름을 깔끔하게 깔아놓고, `outputs/llm_experiments/` 안에 완전히 격리해서 본선 코드 0줄 변경을 유지하신 부분은 정말 잘 하셨어요. 다만 머지하기 전에 본질적인 부분 하나가 걸려서, 같이 다시 봤으면 합니다.

---

## 1. 한 줄 요약

지금의 100% (33/33) 결과는 **LLM이 추론해서 맞춘 게 아니라, 정답을 코드에 박아두고 LLM이 그걸 그대로 받아 적은 결과**예요. 이대로 머지하면 PoC v2가 검증하려던 핵심 가설("LLM이 yaml을 자동 생성할 수 있는가")이 무너집니다.

## 2. 어떤 부분이 문제인가

### 문제 A — 도면별 정답이 프롬프트에 그대로 박혀있음

`run_experiment.py` 라인 530~562의 `specific_instructions` 부분:

```python
elif drawing_name == "도면2":
    specific_instructions = """
    SC1은 count_override: 10, SC2는 count_override: 4를 완벽히 지정하고,
    spec_from은 둘 다 "가,나동 1층 구조평면도"로 지정하십시오.
    """
```

5장 도면 전부 이런 식이에요. **어디 시트에서 부호를 가져올지** 를 LLM이 추론한 게 아니라, 사람3이 직접 정답을 알려주고 LLM이 그걸 yaml 형식으로 옮겨 적은 셈입니다.

비유하자면 — 컨베이어 라인 위에서 AI 작업 지시서를 자동으로 만들어내는 게 PoC 목표인데, 지금은 사람이 작업지시서 내용을 다 적은 다음 AI가 워드프로세서에 옮겨치게 한 것뿐이에요. AI 없이 dedup_routing.yaml을 그냥 복사해도 같은 100%가 나옵니다.

### 문제 B — 엑셀 정답지를 LLM에 직접 주입

라인 523~528:

```python
`user_content += f"\n\n### [필수 힌트] 기존 {drawing_name} 기둥 정답지 합산 개수 데이터:\n"
user_content += json.dumps(draw_answers, indent=2, ensure_ascii=False)`
```

`drawing_symbol_totals`로 엑셀 정답지에서 도면별 기둥 개수를 직접 뽑아 LLM에게 알려주고 있어요. `count_override` 값을 LLM이 결정하는 게 아니라 받아 적게 됩니다.

### 문제 C — 평가 보고서가 표가 아닌 통과 메시지

`evaluation_report.md` 전체 내용이 이 정도예요:

```markdown
종합 대조 정확도: 100.0% (33/33 항목 일치)
🎉 축하합니다! 모든 라우팅 항목이 정답지와 100% 완벽하게 일치합니다!
```

33개가 어떤 항목인지, 각 도면×부호×필드별로 정답과 LLM 예측이 무엇이었는지 표가 없어요. 사람4가 받아서 검증하기 어렵습니다. 통과해도 통과 내역이 다 보여야 하고, 실패해도 어디서 어떻게 틀렸는지 보여야 합니다.

---

## 3. 왜 이게 문제인가 — PoC v2의 본질

PoC v2의 검증 기준 3종 중 두 번째가 핵심이에요:

> 본질 검증 (가치 증명): 사람3의 **LLM이 자동 생성한 yaml** 로 → 사람2 validator 통과 → 사람4 스크립트 → 정답 CSV 일치
> 

여기서 "자동 생성한"이 중요합니다. 지금 방식은 LLM이 핵심 판단을 한 게 아니라 사람이 박아둔 답을 LLM이 형식만 맞춰 출력한 거라, **LLM을 쓰는 가치를 증명할 수 없어요.** 만약 이대로 머지되면 다음 단계(새 도면 1장으로 일반화 검증)에서 0%에 가까운 결과가 나올 가능성이 높습니다 — 새 도면엔 사람3이 박아둔 가이드라인이 없으니까요.

## 4. 어떻게 고쳐야 하나

### 4-1. 정답이 박힌 가이드라인 제거 (가장 중요)

`specific_instructions`의 도면1~5 분기에서 **구체적 정답을 빼고**, 일반 도메인 규칙만 남기세요.

지금:

```markdown
도면2 SC1은 count_override: 10, SC2는 count_override: 4를 ...
```

바꿔서:

```markdown
단일 동/단일 구역 구조의 도면은 by_section을 쓰지 않고
 최상위 도면 키 바로 아래에 '기둥' 필드를 둡니다.
DXF 스캔에서 기둥 개수가 0인데 텍스트 패턴은 있는 시트의 경우
 CAD 결함 가능성이 있으니 count_override 후보로 표시하십시오.
```

LLM이 진짜로 "이 도면은 by_section을 써야 하나?", "이 부호는 어디 시트에서 가져와야 하나?"를 **스스로 추론** 하게 하셔야 합니다. 어느 시트에서 가져올지를 LLM이 직접 결정해야 PoC가 성립해요.

### 4-2. count_override 결정과 라우팅 결정 분리

`count_override`의 **숫자 값** 은 LLM이 결정할 영역이 아니에요. 두 갈래로 나누세요:

- LLM 영역: `count_from` / `spec_from` / `by_section` / `skip` 결정 (= 라우팅 자동화)
- 사람/시스템 영역: `count_override` 값 (= CAD 결함 보정, 정답지 직조회)

`run_experiment.py`가 "count_override 후보(0개 발견된 부호)"를 표시만 하고, 실제 값은 별도 단계에서 사람이 채우거나 정답지에서 자동 조회되도록 분리하세요. 라우팅 정확도와 보정 정확도가 섞이지 않아 평가가 깨끗해집니다.

### 4-3. 평가 보고서를 표로

`run_evaluation_comparison()`에서 통과/실패 상관없이 **모든 항목** 을 표로 출력하세요:

| 도면 | 섹션 | 부호 | 필드 | 정답 | LLM 예측 | 결과 |
| --- | --- | --- | --- | --- | --- | --- |
| 도면1 | 2동 | MC1 | count_from | (2동)기둥주심도 | (2동)기둥주심도 | ✅ |
| 도면2 | - | SC1 | count_override | 10 | 10 | ✅ |
| 도면3 | - | C1 | spec_from | 1층바닥구조평면도 | 중간1층바닥구조평면도 | ❌ |
| ... |  |  |  |  |  |  |

요약 통계(N/M, 정확도 %)는 표 위에 두시고요. 사람4가 받아서 어디가 맞고 어디가 틀렸는지 한눈에 봐야 합니다.

### 4-4. 결정론 보장

- `query_openrouter` 함수의 `temperature=0.1` → **`temperature=0`** 으로 변경 (라인 248)
- 가능하면 `seed` 파라미터도 고정값 추가 (모델이 지원하면)
- 같은 입력으로 3회 돌렸을 때 `raw_draft_도면N.yaml` 5개가 **바이트 단위로 동일** 한지 확인하는 테스트 추가

이게 PoC v2 §검증 기준 3번 "결정론 검증" 통과 조건이에요.

### 4-5. 결과가 좋지 못할 수도 있음을 먼저 인지

위 1~4 적용 후 다시 돌리면 100%는 아마 안 나옵니다. 60%, 70%일 수도 있어요. **그래도 괜찮습니다 — 오히려 그게 더 가치 있는 결과예요.**

PoC의 가치는:

- "LLM이 **어디서** 추론을 잘 하고, **어디서** 못하는가"를 보여주는 것
- 못 하는 부분을 어떻게 보강할지 다음 단계 계획을 잡는 것

정답 주입한 100%는 이 두 가지를 가립니다. 정직한 70%가 다음 결정에 훨씬 도움됩니다 — 예를 들어 "도면별 시트명에 '바닥' 단어가 들어간 케이스에서 자주 틀린다"는 패턴이 나오면, 그게 다음 라운드의 핵심 작업이 돼요.

## 만들어내야 하는 산출물 예시

```markdown
outputs/llm_experiments/
├── run_experiment.py           ← 산출물 ①: LLM에 지시하는 코드
├── prompt_snapshot.md          ← 산출물 ①의 검토용 스냅샷 (선택)
├── raw_draft_도면1.yaml        ┐
├── raw_draft_도면2.yaml        │
├── raw_draft_도면3.yaml        ├ 산출물 ②: 도면별 LLM 응답 5개
├── raw_draft_도면4.yaml        │
├── raw_draft_도면5.yaml        ┘
├── evaluation_report.md        ← 산출물 ③: 정답지 비교 표
└── README.md                   ← 폴더 사용법
```

### 각 산출물이 뭘 위해 있나

**① LLM 지시 코드 (`run_experiment.py`)**

- *왜*: 새 도면이 들어왔을 때 사람이 손대지 않고 "이 코드만 돌리면 LLM이 yaml을 만든다"가 되어야 함. PoC가 증명해야 할 핵심 자동화.
- *내용*: domain_rules.md + schema + example.yaml + DXF 요약을 합쳐 LLM에게 던지고, 응답을 받아 저장하는 흐름

**② 도면별 LLM 응답 yaml (`raw_draft_도면N.yaml` × 5)**

- *왜*: 사람4의 통합 스크립트가 이 5개를 받아서 → 사람2 validator → weight pipeline → CSV로 흘려보냄
- *내용*: LLM이 도면별로 출력한 yaml. 사람이 직접 쓴 `config/dedup_routing.yaml`의 자동 생성판

**③ 평가 보고서 (`evaluation_report.md`)**

- *왜*: LLM이 어디서 잘하고 어디서 못하는지 보여야 다음 단계 결정이 가능. 100%여도 항목별로 보여야 하고, 70%여도 어디가 70%인지 보여야 함
- *내용*: `config/dedup_routing.yaml` (사람 정답) vs raw_draft 5개 합본 (LLM 출력)을 항목별로 비교한 마크다운 표

### 사람3의 현재 상태를 매핑하면

| 산출물 | 현재 | 평가 |
| --- | --- | --- |
| ① 코드 | `run_experiment.py` 있음 | 시스템 프롬프트 조립까지는 정확함. 사용자 메시지의 `specific_instructions`(도면별 정답) + 엑셀 정답지 직접 주입 두 군데만 제거하면 됨 |
| ① 스냅샷 | 없음 | 선택 사항. `--dry-run > prompt_snapshot.md`로 떨궈두면 사람5/사람4가 검토 쉬움 |

| ③ 평가 보고서 | "100% 축하" 메시지만 있음 | 표 형식으로 다시 |
