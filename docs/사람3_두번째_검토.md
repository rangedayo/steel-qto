수고 많으셨어요. 지난번 피드백을 정말 성실하게 반영하셨고, 그 결과 **이번 60.6%는 PoC가 처음으로 진짜 의미 있는 결과를 얻은 순간**이에요. 100% 컨닝 때보다 훨씬 가치 있는 데이터입니다.

본격적으로 머지 가기 전에 같이 한 번 짚어볼 게 있어서 정리합니다.

## 1. 먼저, 100%에서 60.6%로 떨어진 게 왜 좋은 일인가

비유로 풀면 이렇습니다. 공장에서 새 자동화 라인을 설치했어요. 이 라인이 진짜 일을 할 수 있는지 시험을 해봐야 합니다.

- **1차 시험 (100%)**: 작업자에게 정답을 미리 알려주고 옮겨 적게 시킴. → 100% 맞췄지만, 이건 자동화 라인의 실력이 아닙니다.
- **2차 시험 (60.6%)**: 정답 없이 도면만 보고 작업지시서를 만들게 시킴. → 60.6%지만, **이게 진짜 실력입니다.**

60.6%라는 숫자가 처음으로 "이 자동화 라인이 어디까지 할 수 있는지"를 보여줬어요. 어디서 잘하고 어디서 막히는지가 보이니까, **다음에 뭘 보완해야 할지** 가 명확해집니다. 이게 PoC의 진짜 가치예요.

## 2. 이번에 잘 고치신 것들

| 지난번 지적 | 이번에 어떻게 됐나 |
| --- | --- |
| 도면별 정답이 프롬프트에 박혀있었음 | ✅ 다 빼셨고, 일반 도메인 규칙만 남김 |

거기에 **count_override 보정 로직**도 깔끔하게 풀어주셨어요. "LLM은 여기에 override가 필요하다는 판단만 하고, 정확한 숫자는 시스템이 정답지에서 자동 조회"하는 구조 — 좋은 분리입니다.

## 3. 60.6%가 알려주는 것 — 도면별 패턴

각 도면별로 점수를 보면 이렇습니다:

| 도면 | 결과 | 한 줄 평 |
| --- | --- | --- |
| 도면1 | 4/8 (50%) | 시트 이름이 비슷해서 헷갈림 |
| 도면2 | 2/4 (50%) | "이건 도면에 안 그려진 케이스다"를 못 알아챔 |
| 도면3 | 1/8 (**12.5%**) | 너무 복잡해서 부호 4개 중 3개 누락 |
| 도면4 | 4/4 (**100%**) | 간단한 케이스라 완벽 |
| 도면5 | 8/8 (**100%**) | 간단한 케이스라 완벽 |

**무엇을 알 수 있나요?** LLM은 "도면에 평면도 시트가 1장이고 거기서 다 가져오면 되는" 단순한 케이스(도면4·5)에서는 잘 합니다. 그런데 시트 여러 장에서 정보를 골라 조합해야 하는 복잡한 케이스에서 헤맵니다.

### 어디서 막혔는지 자세히

### 도면1 — "부호도"와 "주심도"를 다른 시트로 봄

- **LLM이 한 일**: count_from은 `기둥부호도`에서, spec_from은 `기둥주심도`에서 가져옴
- **정답**: 둘 다 `기둥주심도`에서 가져와야 함
- **LLM의 생각**: 도메인 규칙서에 "본체 카운트와 일람표 규격은 섞지 않는다"고 적혀있으니, count는 "부호가 그려진 시트"에서, 규격은 "주심이 그려진 시트"에서 따로 가져옴
- **진짜 이유**: 이 도면에서는 `주심도` 한 시트가 부호 카운트와 규격을 둘 다 담당함. 그런데 사람1이 작성한 도메인 규칙서에 "주심도 시트가 두 역할을 겸하는 경우"라는 패턴이 명시되어 있지 않음.

→ **사람1 자료가 한 번 더 보강되어야 할 부분**

### 도면2 — CAD에서 부호가 안 잡히는 케이스를 못 알아챔

- **LLM이 한 일**: `count_from: 가나동1층구조평면도`로 처리 (개수를 시트에서 셀 수 있다고 판단)
- **정답**: `count_override: 10`, `count_override: 4` (CAD 결함으로 못 셈 → 정답지 값 사용)
- **LLM의 생각**: 도면 안에 부호가 있다고 보고, 그 시트에서 세면 된다고 판단
- **진짜 이유**: 이 도면은 CAD 파일 자체가 부호를 텍스트가 아닌 다른 방식으로 저장해서, 자동으로 세는 게 불가능함. 그러면 "정답지에서 값을 가져오는 모드"(count_override)로 가야 하는데, LLM이 이 신호를 못 알아챔.

→ **사람1 자료에 "이런 패턴이면 override 써라"가 더 명확해져야 함**

### 도면3 — 부호 4개 중 3개 누락

- **LLM이 한 일**: C1만 출력. C2, C3, C4 완전 누락
- **정답**: C1, C2, C3, C4 모두 출력해야 함
- **LLM의 생각**: 알 수 없음. 도면3은 CAD 파일 구조가 복잡한 케이스라 LLM이 헤맨 듯
- **진짜 이유**: 도면3은 가장 까다로운 유형. LLM이 길을 잃었습니다.

→ **이건 LLM 한계일 가능성이 큼. 다음 라운드(비전 모델 보강 후보) 케이스**

## 4. 마지막으로 짚을 작은 것 4가지

### 4-1. 모델명 한 번 확인 부탁드려요

`.env`에 `MODEL_NAME=deepseek/deepseek-v4-flash`로 되어있는데, 이 모델이 실제로 OpenRouter에 있는지 확인 부탁드립니다. 만약 존재하지 않는 모델명이면 OpenRouter가 다른 모델로 자동 대체했을 가능성이 있어서, 60.6%가 어느 모델의 실력인지 모호해져요.

확인 방법: OpenRouter 사이트에서 모델 목록을 검색해보거나, 한 번 호출했을 때 응답 헤더에 실제 사용된 모델명이 찍히는지 확인해주세요.

### 4-2. prompt_snapshot이 dry-run 때 만들어져서 가짜 데이터로 채워짐

`prompt_snapshot.md`를 보면 도면별 user message 안에 `total_texts: 50, found_symbols 각 5개, height 150~300`이 똑같이 박혀있어요. 이건 dry-run 모드에서 임시로 만든 가짜 데이터입니다 (run_experiment.py 라인 654~658).

실제로 LLM을 호출할 때는 진짜 DXF 분석 데이터가 들어가는데, 사람이 검토할 수 있는 스냅샷은 가짜 데이터로 만들어져있어요.

→ 가능하면 실제 호출할 때도 첫 도면 1회 분의 user message는 파일로 떨궈주시면 사람5가 검토할 때 좋습니다.

### 4-3. 결정론 3회 반복이 항상 돌아요

지금 코드는 `--dry-run`이 아닐 때 무조건 3회 반복합니다 (라인 715). 이러면 API 비용이 3배가 돼요. 개발 중에 자주 돌리면 부담입니다.

→ `--verify-determinism` 같은 별도 옵션으로 분리해주시면, 평소엔 1회만 돌고 검증 필요할 때만 3회 돌게 할 수 있어요.

### 4-4. 사람3이 만든 4개 통합 yaml은 사람4 영역에 가까움

`outputs/llm_experiments/` 안에 만드신 `dedup_routing.yaml`, `length_routing.yaml`, `sheet_name_overrides.yaml`, `symbol_rules.yaml` 4개 파일은 — 도면별 raw_draft 5개를 합쳐서 만든 부산물이에요.

이건 원래 사람4의 통합 스크립트가 할 일에 가까워서, 사람3 산출물에 두면 사람4와 영역이 겹칠 수 있어요. 다만 사람3이 자기 결과를 평가하려면 어쨌든 합쳐서 정답지와 비교는 해야 하니, **평가 목적의 임시 파일**로 두는 건 OK입니다.

→ 사람4와 합쳐 작업할 때 이 부분을 어떻게 정리할지 한 번 협의 필요.

## 5. 다음 단계로 가는 길

지금 60.6%가 PoC의 진짜 시작점이에요. 두 갈래 작업이 가능합니다:

### 갈래 A — 사람1 자료 보강 (도메인 규칙 강화)

`docs/domain_rules.md`에 아래 패턴을 추가하면 60.6% → 70~80%대로 올라갈 가능성이 큽니다:

- "주심도 시트 하나가 부호 카운트와 규격을 둘 다 담당하는 도면 패턴" 명시 (도면1 케이스)
- "DXF 발견 수가 0인데 도면에 부호가 분명히 있는 경우 → count_override 사용" 구체 규칙 (도면2 케이스)
- 도면3 유형의 복잡 케이스 패턴 추가

### 갈래 B — 그래도 안 올라가는 부분이 진짜 가치

도메인 규칙을 보강해도 안 올라가는 부분(특히 도면3)이 결국 **LLM 한계**입니다. 이 부분은 인계 v2 §로드맵 2단계의 비전 모델 보강 후보로 넘어갑니다. 즉 "LLM이 못 하는 게 뭔지" 명확히 짚는 것 자체가 PoC의 결론이에요.
