# steel-qto 팀원 안내 가이드

> 이 문서는 PoC v1 레포(steel-qto)에 팀원이 합류할 때 사용하는 가이드입니다.
> 슬랙·메일로 팀원에게 공유하거나, `docs/팀원_안내.md`로 레포에 같이 올려둘 수 있습니다.

---

## 0. 한눈에 보기

```
팀원 공통 진행 절차      역할별 시작 파일
──────────────────      ────────────────
①  invitation 수락       사람1: domain_rules_seed.md
②  clone + 환경 셋업      사람2: dedup_loader.py
③  pytest 263 PASS 확인   사람3: prompts/llm-wiki
④  각자 역할 파일 읽기    사람4: weight_pipeline.py
⑤  feature 브랜치 작업
⑥  PR 생성·리뷰·머지
```

## 1. 사용자가 먼저 해야 할 일

### 1-1. 팀원 GitHub 계정 추가 (collaborator)

`steel-qto` 가 public이지만, **본 레포에 직접 push하려면 collaborator 권한이 필요**합니다.

```
GitHub → steel-qto → Settings → Collaborators →
  "Add people" 클릭 → 팀원 GitHub username 입력 → Send invitation
```

추가할 팀원:
- [ ] 사람1 담당자 (도메인 규칙 정리)
- [ ] 사람2 담당자 (스키마 설계)
- [ ] 사람3 담당자 (LLM 라우팅 실험)
- [ ] 사람4 담당자 (통합 코드)

### 1-2. Notion 비공개 채널에 DXF 정리

회귀 테스트가 돌려면 `sample_data/*.dxf` 5종이 필요합니다 (기밀이라 깃에 없음).

준비할 것:
- [ ] 도면1.dxf ~ 도면5.dxf (5장)
- [ ] 분리본 dxf 파일들 (Y03 단독본, 종단면도 분리본 등)
- [ ] 팀원이 받을 수 있는 비공개 Notion 채널 또는 사내 스토리지 위치

### 1-3. 안내

```
📐 PoC v1 레포 공유합니다 — https://github.com/rangedayo/steel-qto

[준비물]
- Python 3.11 설치 (필수, 3.12 이상은 호환 미검증)
- Git 설치
- (한글 Windows) PYTHONUTF8=1 환경변수

[시작 절차]
1. GitHub에서 collaborator invitation 이메일 → "Accept"
2. README의 "환경 세팅 4단계" 따라 진행
3. Notion 비공개 채널(<링크>)에서 sample_data/*.dxf 받아 배치
4. pytest -v poc_v2/  →  263 PASS 확인
5. 자기 역할별 시작 파일 (가이드 4번 참고)

[작업 방식]
- main 브랜치 직접 push 금지 (자동 차단됨)
- 새 작업: feature 브랜치 생성 → 작업 → push → PR → 리뷰 → 머지
- 회사명·외부 URL·비밀정보 포함 금지 (public 레포라 영구 노출)

[질문·문제]
- 작업 중 이슈: GitHub Issue 생성
- 큰 결정: 멘토님과 함께 검토
```

## 2. 팀원 공통 진행 절차

### 2-1. invitation 수락

이메일 또는 GitHub 알림으로 받은 collaborator invitation을 "Accept" 클릭.

### 2-2. 환경 셋업 (README 3번과 동일)

```bash
git clone https://github.com/rangedayo/steel-qto.git
cd steel-qto

python -m venv .venv
.venv\Scripts\activate          # Windows
# source .venv/bin/activate     # Mac/Linux
pip install -r poc_v2/requirements.txt
```

⚠️ 한글 Windows에서 pip install 오류 시: `set PYTHONUTF8=1` 입력 후 재시도.

### 2-3. DXF 배치

Notion 비공개 채널(사용자가 안내한 링크)에서 받아 `sample_data/` 폴더에 배치.

### 2-4. 회귀 확인

```bash
pytest -v poc_v2/
```

기대 결과: `263 passed / 2 known-fail`

다른 결과가 나오면:
- `263 passed`보다 적게 통과 → 환경 셋업 문제. 슬랙에 보고
- `failed`가 2개보다 많이 나옴 → DXF 배치 문제일 가능성. Notion 채널 안내 재확인

### 2-5. 각자 역할 파일 읽기 (아래 4번 참고)

### 2-6. 작업 시작

```bash
# main을 최신화하고 새 브랜치 생성
git checkout main
git pull
git checkout -b feature/내이름-작업내용

# 작업…

git add .
git commit -m "feat: 작업 내용 한 줄"
git push origin feature/내이름-작업내용
```

GitHub에서 PR 생성 → 사용자(또는 팀원) 리뷰 받기 → 승인 후 머지.

## 3. 주의사항

### 절대 금지

- ❌ **main에 직접 push** — branch protection이 자동 차단하지만, 시도 자체를 하지 않기
- ❌ **`sample_data/*.dxf`를 git에 추가** — `.gitignore`로 이미 차단되지만 명시적으로 git add 하지 않기
- ❌ **회사명·고객사명·이메일·API 키·토큰** 등 포함 금지 — public 레포라 영구 노출
- ❌ **Python 3.12 이상 사용** — 호환성 미검증 (3.11.x만)
- ❌ **본선 코드·yaml·정답지 무단 수정** — 회귀가 깨지면 PR 거부 사유

### 권장 사항

- ✅ Python **3.11.x** 사용 (3.11.9 권장)
- ✅ 한글 Windows에서 `set PYTHONUTF8=1` 환경변수
- ✅ PR 제목: `feat: ...`, `fix: ...`, `docs: ...` 같은 prefix 사용
- ✅ PR 본문에 "무엇을·왜·어떻게" 3줄 요약
- ✅ 회귀 테스트 통과 확인 후 PR 올리기 (`pytest -v poc_v2/`)

## 4. 역할별 시작 파일

각 팀원이 자기 역할에 맞게 어떤 파일부터 읽고 무엇을 만들지 정리했습니다.

### 4-1. 사람1 — 도메인 규칙 정리

> **역할 비유**: 도면 읽기 선생님. LLM이 도면 보는 법을 배우는 교재를 만듭니다.

**먼저 읽을 파일** (순서대로):

| 순번 | 파일 | 왜 |
|---|---|---|
| 1 | [`docs/domain_rules_seed.md`](docs/domain_rules_seed.md) | **본 작업의 시드**. 풀어쓸 도메인 규칙 초안 |
| 2 | [`docs/round_history.md`](docs/round_history.md) | 14개 라운드 결정 흐름 — 왜 이 규칙들이 나왔는지 |
| 3 | `config/symbol_rules.yaml` | 부호 화이트리스트 규칙 (자동 제외 카테고리 등) |
| 4 | `config/dedup_routing.yaml` | **가장 중요한 결정 파일** — 중복 판별 |
| 5 | `reference_materials/도면_정답지.xlsx` | 정답 데이터 (카운트) |
| 6 | `reference_materials/도면_길이_정답지.xlsx` | 정답 데이터 (길이·규격) |

**참고 자료**:
- `outputs/round_*_보고서.md` — 각 라운드 의사결정 기록
- `outputs/prompts/llm-wiki_prompt.md` — 이미 작성된 도메인 룰북 명세

**산출물**:
- 📘 `docs/domain_rules.md` (확장본) — domain_rules_seed.md를 다듬어 완성도 ↑
- 📄 `docs/example.yaml` — **도면1 모범답안 yaml** (도면2~5 LLM 처리 시 본보기)

**멘토와 상의할 일이 가장 많은 자리** — 적산 도메인 질문은 멘토에게.

---

### 4-2. 사람2 — 스키마 설계

> **역할 비유**: 양식지 디자이너. LLM이 yaml 쓸 때 따라야 할 빈 양식을 만듭니다.

**먼저 읽을 파일** (순서대로):

| 순번 | 파일 | 왜 |
|---|---|---|
| 1 | `config/dedup_routing.yaml` | **스키마 ground truth**. 사람이 작성한 정답 yaml |
| 2 | `config/length_routing.yaml` | 보조 스키마 |
| 3 | `poc_v2/qto/dedup_loader.py` | 현재 yaml 파싱·검증 로직 (스키마 검증의 시작점) |
| 4 | `outputs/round_weight1a_보고서.md` | 1a 시점 스키마 |
| 5 | `outputs/round_weight1b_보고서.md` | 1b에서 확장된 스키마 (`by_section` / `skip` / `count_override`) |

**산출물**:
- 📐 `routing.schema.json` (jsonschema 표준)
- 🔍 검증 코드 (사람4가 사용할 jsonschema 라이브러리 호출)

**⏰ 가장 먼저 끝내야 하는 자리**: 사람1·3·4가 모두 사람2의 스키마를 보고 일합니다.
**Day 1·2 안에 끝나야 나머지가 움직일 수 있어요.**

---

### 4-3. 사람3 — LLM 라우팅 실험

> **역할 비유**: AI에게 일 시키는 사람. LLM에게 정확한 명령을 내려 yaml 초안을 받아냅니다.

**입력 자료** (사람1·2의 산출물):
- 사람1의 `docs/domain_rules.md`, `docs/example.yaml`
- 사람2의 `routing.schema.json`

**먼저 읽을 파일**:

| 순번 | 파일 | 왜 |
|---|---|---|
| 1 | `config/dedup_routing.yaml` | **사람이 작성한 정답 yaml** — LLM 출력 평가용 ground truth |
| 2 | `outputs/prompts/llm-wiki_prompt.md` | 이미 작성된 도메인 룰북 (LLM에 넣을 자료) |
| 3 | `docs/round_history.md` | 어떤 결정들이 yaml에 반영됐는지 맥락 |
| 4 | `reference_materials/도면_정답지.xlsx` | 정답 |
| 5 | `sample_data/*.dxf` | 도면 원본 데이터 |

**산출물**:
- 💬 LLM 프롬프트 (사람1 자료 + 사람2 스키마 + few-shot 예시)
- 📂 도면별 LLM 출력 yaml
- 📊 평가 결과 (LLM yaml vs 사람 yaml 비교 표)

**⚠️ 우선순위**: 도면 5장 동시가 아니라 **도면 한 장이라도 끝까지 작동시키기**가 첫 목표.
시간 예측이 가장 어려운 자리라 안전한 첫 목표 설정 중요.

---

### 4-4. 사람4 — 통합 코드

> **역할 비유**: 안전 검사관. "LLM이 실수할 수 있다는 가정"으로 일합니다.

**먼저 읽을 파일** (순서대로):

| 순번 | 파일 | 왜 |
|---|---|---|
| 1 | `poc_v2/qto/weight_pipeline.py` | **현재 yaml → CSV 흐름의 코어**. 어떻게 yaml을 받아 곱하는지 |
| 2 | `poc_v2/qto/dedup_loader.py` | yaml 파싱 로직 (사람2 스키마 검증으로 교체 대상) |
| 3 | `poc_v2/qto/export_weight_csv.py` | CLI 진입점 (--drawing, --all 옵션 처리) |
| 4 | 사람2의 `routing.schema.json` | 검증 호출 대상 |

**참고 자료**:
- `outputs/handoff_task1_보고서.md` — 환경 재현성 검증 패턴
- `outputs/handoff_task2_보고서.md` — 정제 작업 패턴
- `poc_v2/tests/` — 기존 테스트 구조 참고 (자기 작업에도 회귀 추가)

**산출물**:
- 🛡️ **end-to-end 메인 스크립트**:
  1. 사람3의 LLM 출력 yaml 받기
  2. 사람2의 스키마로 형식 검증
  3. PR 처리된 yaml로 weight_pipeline → CSV 출력

**원칙**: **결정론적 코드** (같은 입력 → 같은 출력). LLM처럼 들쭉날쭉 X.
도메인 지식·LLM 지식 없이 파이썬 코드만 잘 짜면 되는 자리.

## 5. AI를 활용해도 되나요?

업로드된 분담 문서의 톤을 보면 "각 팀원이 AI랑 대화하며 자기 역할을 풀어내는 것"도 좋은 옵션이에요.

추천 활용 방식:

| 상황 | AI 활용 |
|---|---|
| 코드 이해 막힘 | "이 파일이 무엇을 하는지 설명해줘" |
| 작업 시작 | "내 역할은 X, 시작할 첫 PR을 어떻게 잡으면 좋을까?" |
| 막힐 때 | "이 에러 메시지 뭔지, 어떻게 해결하지?" |
| PR 작성 | "이 변경사항에 맞는 PR 제목·본문 써줘" |

⚠️ 단, **회사 기밀 정보(고객사·도면 데이터)는 AI에 그대로 붙여넣지 말기.** 공개 AI 서비스에 들어간 정보는 학습 데이터로 사용될 가능성이 있어요.

## 6. 첫 PR 추천 (워밍업)

각 팀원이 첫 PR로 만들기 좋은 작은 작업:

| 팀원 | 워밍업 PR |
|---|---|
| 사람1 | `docs/domain_rules_seed.md`에 자기 이해 메모 한 단락 추가 |
| 사람2 | `routing.schema.json` 빈 파일 + 첫 필드(`drawing`) 만 정의 |
| 사람3 | `outputs/llm_experiments/` 폴더 신설 + README 한 줄 |
| 사람4 | `poc_v2/qto/` 코드 읽고 흐름 다이어그램(주석)으로 PR |

이렇게 작은 PR을 한 번 거치면:
1. 환경 셋업이 진짜로 됐는지 검증
2. PR 워크플로우 한 번 경험
3. 코드베이스 한 부분이라도 직접 만져본 상태로 시작

